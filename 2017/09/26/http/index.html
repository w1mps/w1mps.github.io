关于HTTP

发表于 2017-09-26 | 分类于 web

HTTP

HTTP（HyperTextTransferProtocol）是超文本传输协议的缩写，它用于传送WWW方式的数据，关于HTTP 协议的详细内容请参考RFC2616。

HTTP是一个客户端和服务器端请求和应答的标准（TCP）。客户端是终端用户，服务器端是网站。HTTP客户端发起一个请求，建立一个到服务器指定端口（默认是80端口）的TCP连接。HTTP服务器则在那个端口监听客户端发送过来的请求。一旦收到请求，服务器则发回一个状态行，比如”HTTP/1.1 200 OK”，和其他一些响应返回的消息，消息体可能包含的是客户端所请求的文件或者错误消息、亦或者其它一些信息。

HTTPS

HTTPS是具有安全性的SSL加密传输协议，简单的来说就是HTTP+SSL，相当于是更加安全版的HTTP，但是HTTP的连接的默认端口是80，而HTTPS则是443。

HTTP请求头

标准HTTP头

Accept

告诉WEB服务器自己接受什么介质类型，/ 表示任何类型，type/* 表示该类型下的所有子类型，type/sub-type。

Accept-Charset

浏览器申明自己接收的字符集

Accept-Encoding

浏览器可以处理的编码方式，注意这里的编码方式有别于字符集，这里的编码方式通常指gzip,deflate等。

Accept-Language

浏览器接收的语言(例如：Accept-Language: zh-CN)

Accept-Ranges

WEB服务器表明自己是否接受获取其某个实体的一部分（比如文件的一部分）的请求。

  • bytes：接受
  • none：不接受

Age

当代理服务器用自己缓存的实体去响应请求时，该头部表明该实体从产生到现在经过了多长时间。

Authorization

当客户端接收到来自WEB服务器的 WWW-Authenticate 响应时，用该头部来回应自己的身份验证信息给WEB服务器。

Expect

客户端使用该字段告知服务器，期望出现某种特定行为，如果服务器无法理解期望作出回应而发生错误，会返回状态码417Expect Failed

Cache-Control

用于指定所有缓存机制在整个请求/响应链中必须服从的指令。
请求头里的取值有：

  • no-cache：必须先与服务器确认返回的响应是否被更改，然后才能使用该响应来满足后续对同一个网址的请求。因此，如果存在合适的验证令牌 (ETag)，no-cache 会发起往返通信来验证缓存的响应，如果资源未被更改，可以避免下载。
  • max-age:指示客户机可以接收生存期不大于指定时间（以秒为单位）的响应。
  • max-stale:指示客户机可以接收超出超时期间的响应消息。如果指定max-stale消息的值，那么客户机可以接收超出超时期指定值之内的响应消息。。
  • min-fresh:指示客户机可以接收响应时间小于当前时间加上指定时间的响应。

请求头和响应头都有的取值：

  • no-store:所有内容都不会被缓存到缓存或 Internet 临时文件中。

Connection

告诉服务器这个user agent（通常为浏览器）想要使用怎样的连接方式。http1.1默认是keep-alive。

  • keep-alive：是指浏览器和服务器之间的通信连接会被持续保存，不会马上关闭。
  • close：是指会在response后马上关闭。

Date

Date头域表示消息发送的时间，时间的描述格式由rfc822定义。

Cookie

浏览器向服务器发送请求时发送cookie，或者服务器向浏览器附加cookie，就是将cookie附加在这里的。

Content-MD5

使用base64进行了编码的请求体的MD5校验和。

Content-Type

请求体中的内容的mime类型。例：Content-Type: application/x-www-form-urlencoded

Content-Length

一个请求的请求体的内存长度，单位为字节(byte)。请求体是指在HTTP头结束后，两个CR-LF字符组之后的内容。

From

发送这个请求的用户的email地址。

Host

目标服务器的域名或IP地址，如果不是通用端口，还包含该端口号。

ETag

供 WEB 服务器判断一个对象是否改变了。如前一次请求某个html文件时，获得了其ETag，当这次又请求这个文件时，浏览器就会把先前获得的ETag值发送给WEB服务器，然后WEB服务器会把这个ETag跟该文件的当前ETag进行对比，然后就知道这个文件是否改变。

If-Match

只有请求内容与实体相匹配才有效，即对象的ETag没有改变，其实也就意味著对象没有改变，才执行请求的动作。

If-None-Match

当请求内容与实体不匹配时有效，即对象的ETag改变了，其实也就意味著对象也改变了，才执行请求的动作。如果其内容未改变，则返回304告诉浏览器使用其本地的缓存。

If-Modified-Since

询问服务器现在正在请求的资源在某个时间以来有没有被修改过，如果没有，服务器则返回304状态来告诉浏览器使用浏览器自己本地的缓存，如果有修改过，则返回200，并从服务器发送新的资源。

If-Unmodified-Since

询问服务器现在正在请求的资源在某个时间以来有没有被修改过，如果没有，则返回200，进行传输，如果有，则不进行传输，返回412 Precondition Failed作为响应返回。

If-Range

如果实体未改变，服务器发送客户端丢失的部分，否则发送整个实体，参数也为Etag。

Pragma

用来包含实现特定的指令。最常用的是Pragma:no-cache。在HTTP/1.1协议中，它的含义和Cache-Control:no-cache相同。

Max-Forwards

通过指定Max-Forwards的取值，每经过一次转发，值-1，直到为0，直接返回响应。

Proxy-Authorization

同Authorization是服务器用来验证身份信息的，不过这里是指代理服务器。

Range

使用Range来获取服务器上的指定部分的资源，即可告知服务器资源的指定范围。
接收到附带Range字段的服务器，会返回206 Partial Content的响应；当其无法处理请求时，则返回200 OK的响应及全部资源。

Referer

告知服务器是从哪个url而来，即表明来源url。

User-Agent

包含了浏览器自身的一些信息，告知服务器是用何浏览器在进行访问。

TE

客户端愿意接受的传输编码，并通知服务器接受接受尾加头信息

Via

用于记录一个请求经过了哪些代理和网关才传输至服务器上。

Warning

记录一些警告信息。

非标准但通用HTTP头

X-Requested-With

用于识别ajax请求，很多javascript框架会发送这个头域。

X-Forwarded-For

记录一个请求从客户端出发到目标服务器过程中经历的代理，或者负载平衡设备的IP。

X-Forwarded-Proto

记录一个请求一个请求最初从浏览器发出时候，是使用什么协议。

DNT

DO NOT TRACK的缩写，要求服务器程序不要跟踪记录用户信息。

  • DNT: 1 (开启DNT)
  • DNT: 0 (关闭DNT)

HTTP响应头

标准HTTP头

Accept-Ranges

表明服务器是否支持指定范围请求及哪种类型的分段请求，如bytes。

Age

当代理服务器用自己缓存的实体去响应请求时，用该头部表明该实体从产生到现在经过多长时间了（单位：秒）

Allow

服务器支持哪些请求方法（如GET、POST等）

Cache-Control

用于指定所有缓存机制在整个请求/响应链中必须服从的指令。
响应头里的取值有：

  • public：所有内容都将被缓存(客户端和代理服务器都可缓存)。
  • private：内容只缓存到私有缓存中(仅客户端可以缓存，代理服务器不可缓存)。
  • no-cache：必须先与服务器确认返回的响应是否被更改，然后才能使用该响应来满足后续对同一个网址的请求。因此，如果存在合适的验证令牌 (ETag)，no-cache 会发起往返通信来验证缓存的响应，如果资源未被更改，可以避免下载。
  • max-age:指示客户机可以接收生存期不大于指定时间（以秒为单位）的响应。

Content-Encoding

web服务器支持的返回内容压缩编码类型。

Content-Language

用于表明响应体所用的语言

Content-Length

用于表明响应体的长度，以字节方式存储的十进制数字来表示，也就是一个数字字符占一个字节，用其对应的ASCII码存储传输。

Content-Location

请求资源可替代的备用地址

Content-MD5

使用base64进行了编码的资源的MD5校验和。

Content-Range

表明在整个返回体中本部分的字节位置

Content-Type

返回内容的MIME类型，例如Content-Type: text/html; charset=utf-8

Date

记录原始服务器消息发出的时间

ETag

请求变量的实体标签的当前值

Expires

表明此次响应过期的日期和时间

Last-Modified

告知浏览器其请求资源的最后修改时间，客户可以通过If-Modified-Since请求头提供一个日期，该请求将被视为一个条件GET，只有改动时间迟于指定时间的文档才会返回，否则返回一个304（Not Modified）状态。

Location

用于重定向接受者到一个新的位置。

Pragma

包括实现特定的指令，它可应用到响应链上的任何接收方

Refresh

应用于重定向或一个新的资源被创造，在5秒之后重定向，
Refresh: 5; url=
http://www.zcmhi.com/archives/94.html

Server

web服务器软件名称

Set-Cookie

设置和页面关联的Cookie。

Transfer-Encoding

消息首部指明了将 entity 安全传递给用户所采用的编码形式。常用的指令有

  • chunked：数据以一系列分块的形式进行发送，Content-Length在这种情况下不被发送。
  • compress：采用 Lempel-Ziv-Welch (LZW) 压缩算法。目前已被大部分浏览器弃用。
  • deflate：采用zlib结构(在RFC1950中规定)，和deflate压缩算法(在RFC1951中规定)。
  • gzip：表示采用Lempel-Ziv coding(LZ77)压缩算法，以及32位CRC校验的编码方式。出于兼容性的考虑， HTTP/1.1 标准提议支持这种编码方式的服务器应该识别作为别名的 x-gzip 指令。
  • identity：用于指代自身。

Vary

告诉下属代理是使用缓存响应还是从原始服务器请求

Via

告知客户端响应从服务器是经过了哪些代理或网关才到达客户端的

WWW-Authenticate

告知客户端应该在Authorization头中提供什么类型的授权信息

Warning

记录一些警告信息。

HTTP状态码

HTTP/1.1中定义了5类状态码， 状态码由三位数字组成，第一个数字定义了响应的类别：

  • 1xx 提示信息          表示请求已被成功接收，继续处理
  • 2xx 成功                 表示请求已被成功接收，理解，接受
  • 3xx 重定向              要完成请求必须进行更进一步的处理
  • 4xx 客户端错误       请求有语法错误或请求无法实现
  • 5xx 服务器端错误    服务器未能实现合法的请求

常见的状态码：

  • 200 OK                             客户端请求成功
  • 400 Bad Request             由于客户端请求有语法错误，不能被服务器所理解。
  • 401 Unauthonzed            请求未经授权。这个状态代码必须和WWW-Authenticate报头域一起使用
  • 403 Forbidden                 服务器收到请求，但是拒绝提供服务。服务器通常会在响应正文中给出不提供服务的原因
  • 404 Not Found                 请求的资源不存在，例如，输入了错误的URL
  • 500 Internal Server Error  服务器发生不可预期的错误，导致无法完成客户端的请求。
  • 503 Service Unavailable   服务器当前不能够处理客户端的请求，在一段时间之后，服务器可能会恢复正常。
